          Feature summary 

          The Persistent WebApp (PWA) feature allows admins to deploy a full screen web app on a Cisco Webex Room Navigator that is located inside or outside of the meeting room. This Web App cannot be dismissed and will be visible at all times. In simple terms, the Room Navigator opens a web app, additionally it opens a websocket channel that allows jsxapi to be available for the web app directly. (same syntax as the embedded macro framework for interacting with the xAPI if you are familiar with that). 

           

          The web app can then interact directly with a limited set of xAPI commands, status and configurations to create custom behavior. How this looks and works is entirely up to the developer of the web app. 

           

           

          Feature description, functional overview, compatibility 

          Purpose

          Allow integration developers to create a custom user interface for their Cisco Webex Room Navigator in form of a web app which is a very popular and easy way of creating good looking user interfaces without needing to have deep knowledge on programming. 

           

          I have been playing around with the feature and created.. at least something.. that should provide a level of understanding on how this feature works. 

           

          So, if you look at the screenshot above, this is actually how my UI looks on my Room Navigator right now.  In the functional overview settings we will go through the feature step by step. This is truly an amazing feature that really open up possibilities for integrating a customized booking experience tailored for the company that deploys it. 

           

          Functional overview

           

          Requirements

          First of all, this feature is supported by the Cisco Webex Room Navigator and in additional to an existing controller (at least one touch controller must be paired). 

           

          If you plan to set this up, and you have a Touch 10 and a Room Navigator, make sure you set up the device with the Touch 10 first (use as the codec controller) and register your device to Webex. If you have two Room Navigators or more, just use the first one as a controller and setup the rest accordingly. 

           

          Now, lets setup the peristent WebApp requirements on the codec before we pair the additional Room Navigator. 
          If you pair the additional Webex Room Navigator before configuring the Persistent WebApp Settings on the codec, you have to factory reset the Navigator and pair it again.

           

          Due to a limitation (pre-GA) you have to set the following configuration, this is because the websocket communication only have support for HTTP at the current development stage and should only be used for testing. HTTPS support is being implemented. 

           

           

           

           

           

           

          xconfiguration NetworkServices HTTP Mode: HTTP+HTTPS

           

           

           

           

           

           

          Next you have to make sure that Websocket is properly configured on the device:

           

           

           

           

           

           

          xconfiguration NetworkServices Websocket: FollowHTTPService

           

           

           

           

           

           

          Now you can punch in the URL to the web app you want to display on the Room Navigator after it is setup:

           

           

           

           

           

           

          xconfiguration UserInterface HomeScreen Peripherals WebApp URL: <URL>

           

           

           

           

           

           

          This will load the web app on the Touch panel. This can be any kind of web page that has been created by the customer or a third party service, like Appspace, for signage or interactive booking: 

           

          These kinds of web apps may do stuff with third party services but they may not have direct access to the xAPI. The "service" may have logic for updating the device through its xAPI over other paths, like shell, cloud xapi or even the XMLAPI - but with very little effort you can have one web app to rule them all. This means you do not have to write additional code to adapt to a specific device. 

           

          You may have, let's say 5 meeting rooms. They all have the same setup, but you dont want to write code for each of these to talk to a booking service. So, the web app works as a more centralized app for utilizing whatever "customer created  service" they have. All of the 5 meeting rooms access the same URL and adapt to each device - displaying the device name on the Room Navigator since its grabbing the data from the xAPI of the device that is accessing the web app. Genius, right?!

           

          So, yes - let's talk about getting the WebApp access to the xAPI. This may sound a bit strange or even "is it really secure?". Well, it depends, but we have strict requirements in order to expose the xAPI and the available commands you can run is controlled with an ACL list (which means you dont have access to the full xAPI). Below is the current ACL list that the persistent web app can access.

          If the WebApp in question does anything malicious the customer has consciously allowed the webapp to get access to the xAPI. Its not like you can end up on a website that can take control over the xAPI without being allowed to do so. This brings us to the first config on the step to allow xAPI access: 

           

           

           

           

           

          xConfiguration WebEngine Features Xapi Peripherals AllowedHosts Hosts: <list of domain names to be matched>

Example:
xConfiguration WebEngine Features Xapi Peripherals AllowedHosts Hosts: "example.com" //Matches example.com hostname
xConfiguration WebEngine Features Xapi Peripherals AllowedHosts Hosts: "example.com example2.com" //Matches the hostnames listed any of the two
xConfiguration WebEngine Features Xapi Peripherals AllowedHosts Hosts: "*" //Matches any hostname

           

           

           

           

           

          Now we need to enable the API key, this is necessary for the web app to get access to the xAPI.

           

           

           

           

           

          xconfiguration Security XAPI WebSocket APIKey Allowed: True

           

           

           

           

           

          Now we can pair the Room Navigator and set it up in Persistent WebApp Mode. Pairing is done in the usual way, when you register the device to Webex you may need to create a user on the device to pair a second device. Also check out the new PIN Pairing feature NPI.

           

          NOTE: YOU MUST PAIR THE ROOM NAVIGATOR OVER THE NETWORK, DIRECT PAIRING IS NOT SUPPORTED WITH PWA

           

          You will get a few options on how you want to deploy your Room Navigator after the device is paired. Select the Persistent WebApp. 

           

           

          If the device is setup correctly, it will now go to the URL you have specified in the configuration and display it in fullscreen. Here is a summary: 

           

           

           

           

           

          # Pre-Requisites
# Must be registered to WebEx
# Must have at least one controller paired (Touch 10 or Room Navigator)
 
### Codec settings
# Enable web services for Websocket XAPI
xconfiguration NetworkServices HTTP Mode: HTTP+HTTPS (pre-GA only)
xconfiguration NetworkServices Websocket: FollowHTTPService
 
# Launch Web App on Navigator
xconfiguration UserInterface HomeScreen Peripherals WebApp Allowed: True // feature toggle (internal) cloudvisible=true (pre-GA only)
xconfiguration UserInterface HomeScreen Peripherals WebApp URL: <URL> // (pubic-api-preview)
 
# Domain name match check for URL entered
xConfiguration WebEngine Features Xapi Peripherals AllowedHosts Hosts: <list of domain names to be matched>
 
# Enable JSXAPI over websocket for use within persistent web app
xconfiguration Security XAPI WebSocket APIKey Allowed: True // public-api-preview, cloudvisible=true
 
### Navigator Settings
# After configuring codec, pair a second navigator over IP
# Select "Persistent Web App" option on out of box screen

The file should be placed on a web server (of which is allowed with the hostname). It has been used with a Room Kit paired with a Touch 10 and a Room Navigator in persistent web app mode. It assumes that all the RoomAnalytics features is enabled on the codec. Feel free to use the code as you want.

<script type="text/javascript">
(function () {
  //Check if the getXAPI module is available
  async function main() {
    if (typeof getXAPI !== 'function') {
      document.body.innerHTML = '<h1 id="notavailable">XAPI is not available!</h1>';
      return;
    }
    
    //Define led colors based on booking availability
    const ledColors = {
           "Free":"green",
           "BookedUntil": "red",
           "FreeUntil": "yellow"
       }
    
    //Define xapi
    const xapi = await getXAPI();
    
    //Declare initial values on load

      deviceType,
      serialNumber,
      presence,
      bookings,
      peopleCount,
      peopleCapacity,
      bookingStatus,
      bookingAvailability,
      temp
    ] = await Promise.all([ 
      xapi.Status.SystemUnit.ProductId.get(),
      xapi.Status.SystemUnit.Hardware.Module.SerialNumber.get(),
      xapi.Status.RoomAnalytics.PeoplePresence.get(),
      xapi.Command.Bookings.List(),
      xapi.Status.RoomAnalytics.PeopleCount.Current.get(),
      xapi.Status.RoomAnalytics.PeopleCount.Capacity.get(),
      xapi.Status.Bookings.Availability.Status.get(),
      xapi.Status.Bookings.Availability.TimeStamp.get(),
      xapi.Status.Peripherals.ConnectedDevice.RoomAnalytics.AmbientTemperature.get()
    ]); 
  
    //Select current led color
    const currentLedColor = ledColors[bookingStatus]
    
    //Function to select new LED colors based on the booking status
    async function led() {
        var bookedStatus = await xapi.Status.Bookings.Availability.Status.get();
        var bookedAvailability = await xapi.Status.Bookings.Availability.TimeStamp.get();
        document.getElementById('led').style["background-color"] = ledColors[bookedStatus];
        document.getElementById('aStatus').innerHTML = bookedStatus;
        document.getElementById('aAvail').innerHTML = bookedAvailability;
    }
    
    //Function to book the device for 30 minutes and print the result of the booking on the screen
    async function book() {
        result = await xapi.Command.Bookings.Book();
        rs = document.getElementById('bookingResult');
        rs.innerHTML = JSON.stringify(result);
        rs.style['display'] = "block"; 
        setTimeout(function(){rs.style['display'] = "none"}, 3000);
    }
    
    async function refreshAnalyticsValues() {
        const [
          pres,
          peopleC,
          tempe
        ] = await Promise.all([ 
          xapi.Status.RoomAnalytics.PeoplePresence.get(),
          xapi.Status.RoomAnalytics.PeopleCount.Current.get(),
          xapi.Status.Peripherals.ConnectedDevice.RoomAnalytics.AmbientTemperature.get()
        ]);
        
        document.getElementById('ambtemp').innerHTML = tempe;
        document.getElementById('count').innerHTML = peopleC;
        document.getElementById('presence').innerHTML = pres;
        led();
    }
  
    
    //Listen for status changes, updating the value for peoplecount when it changes on the codec
    xapi.Status.RoomAnalytics.on((event) => {
        refreshAnalyticsValues();
    });
    xapi.Status.Bookings.on((event) => {
        refreshAnalyticsValues();
    });

                    
    document.body.innerHTML = `
<div class="jumbotron text-center" id="led" style="background-color: ${currentLedColor}">

                    <p class="card-text">Capacity: <b id='relativeHum'>${peopleCapacity}</b></p>

Its using the booked status as a LED indication. You can change the color of the actual LED on a wallmounted Room Navigator in PWA depending on busy state. As I did not have this, I created a HTML led strip that changes based on the Free/Busy state of the codec. It also displays how many people are in the room, the room temperature and when the room becomes free again (if it is busy). Its a simple app that hopefully can help understand the feature better.

          Security

          The Persistent WebApp feature is disabled by default.

          Requires that the admin configures the device to allow the navigator to access WebApps

          The WebApp xAPI access can be restricted to certain domain only.

          The WebApp xAPI access must be stricly enabled from the xAPI by an admin, it is not enough to allow one or more hosts - this comes in addition. 

           

           

           

          Known Issues & Limitations

          • HTTPS traffic not currently enabled between codec and navigator for XAPI communication. This is currently over HTTP and will be switched to HTTPS before the feature is Generally Available. 
          • Must factory reset navigator if paired before setting up the persistent web app settings on the codec
          • Few error handling scenarios need to be addressed. "Out of service" screen when URL is not configured. Fixed
          • Direct pairing is not supported and probably will not be - Network pairing is required
          • The Room Navigator will not go into standby when it is in Persistent WebApp Mode (PWA) - This is by design

           

          Troubleshooting

          If a customer has issues with this feature, make sure that they comply with all the required configurations. 

           

          If all the configurations are OK and there is no obvious reasons why it does not work, it is likely that the issue is caused by their code. We do have a remote debugging framework that allows the customer to connect to their device and see what is happening on the Room Navigator when the page is loading. 

           

          On the codec, enable the following: 

           

           

           

           

           

          xConfiguration WebEngine RemoteDebugging: On

           

           

           

           

           

          On the Room Navigator you will see a red banner with a URL that you can connect to in order to see the session data. 

           

          In a regular browser (for example. Chrome) on your PC/laptop with network access to the Navigator, access the URL in the banner.

           

          You will be met with a list of inspectable pages (normally just one) 

          When you click it you will open up the console of the Room Navigator which will allow you to see all the events that has happened since the current page loaded. If you reload the page on the Room Navigator the inspection session will be broken and you have to access the new session (on the same initial page). 

           

          In the console you will see error messages thrown by the jsxapi websocket in order to identify syntax errors or errors with the xAPI syntax, for example that you are trying to access an xAPI node that is not part of the ACL list. Or if you have typed the command wrong, for example. This is THE troubleshooting tool for developers of apps for the xAPI Web Integration. The debugging can be enabled by the customer and used by the customer during development of their app. 

           

          TAC should also refer the customer to use the remote debugger if they have issues with their App code to identify coding issues or debugging unexpected behavior. This tool has been around since we introduced the webengine on the codec and can also be used for troubleshooting signage issues or other issues on the rendeing of the web app. 

           

          Below is an example on how the remote debugging looks like, from the console view we can see the communication (requests and responses) on the websocket connection to the xAPI on the codec. One of the commands in this case is written incorrectly, and the log is showing me that this issue occurs on line 40 in the app code (xapi.php) that was fetched from the remote website. (My web app). So now I can go and check it out and fix it, try again, etc. 

           

          Cisco TAC do not support troubleshooting third party apps

          We do not support customer created third party apps that are accessed by our devices. This responsibility and support is the author / team / company that has created the app. This means that we do not have to debug their code written for the interaction with the xAPI. 

           

          The reason for this is that it could be syntax errors, or for other reasons, poorly written code that just do not work. For cases like this, please politely refer to the Cisco development community for assistance. There are many people (customers, partners and Cisco employees) that are dealing with integrations and programming towards our devices that wants to help other succeed. Public webex teams space for integration development: https://eurl.io/#rkp76XDrG
